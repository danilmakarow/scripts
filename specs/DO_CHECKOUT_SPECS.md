# docheckout

A CLI for one-shot "stash my work to a new branch and push it" with a
Conventional Commits message generated by Claude Haiku.

Source lives at `src/do-checkout.ts`. After running `pnpm run pack <dirname>` it
is installed as the zsh alias `docheckout`.

## Features

- **Up-front validation**: cwd must be a git repo with pending changes.
- **Branch-aware behavior**:
  - On `master`/`main`: the model is asked for both a branch name and a commit
    message; the script creates the branch, commits, and pushes with `-u`.
  - On any other branch: the script just commits + pushes to that branch.
- **AI-drafted messages**: Claude Haiku
  (`claude-haiku-4-5-20251001`) writes a Conventional Commits subject + body
  from the staged diff.
- **Pretty UX**: shared logger + spinner + 3-line streaming tail of subprocess
  output for every git step (`git add`, `git checkout`, `git commit`, `git push`).

## Setup

1. Copy `.env.example` to `.env` in the project root and add an Anthropic API
   key:

   ```bash
   cp .env.example .env
   echo 'ANTHROPIC_API_KEY=sk-ant-...' >> .env
   ```

2. Install deps and pack the scripts:

   ```bash
   pnpm install
   pnpm run pack <install-dirname>
   ```

   Re-source `~/.zshrc` (or open a new shell) to pick up the `docheckout` alias.

## Usage

```bash
docheckout
```

There are no positional arguments. The script always operates on the current
working directory.

## Env

| Variable | Required | Purpose |
|----------|----------|---------|
| `ANTHROPIC_API_KEY` | yes | Used to call Claude Haiku for the commit message (and branch name when on master/main). |

## Behavior — master / main

When the current branch is `master` or `main`:

1. `git add -A` to stage everything.
2. The staged diff is sent to Claude Haiku with a system prompt enforcing
   Conventional Commits and a user prompt requiring JSON of shape
   `{ "branchName": "...", "commitMessage": "..." }`.
3. The branch name is validated against `^[a-z]+\/[a-z0-9-]+$` and the leading
   `<type>` is expected to mirror the conventional commit type.
4. `git checkout -b <branchName>` creates the new branch.
5. `git commit -F <tmpfile>` records the commit.
6. `git push -u origin <branchName>` pushes and sets upstream.

### Branch-name collision policy

If the suggested branch name already exists locally, the script appends a
numeric suffix (`-2`, `-3`, …) up to `-10` and uses the first free slot. It
emits a warning when a fallback is used. If all 10 candidates are taken, it
fails with a `SuggestionError` listing every name it tried — the user is
expected to either delete the stale branch or re-run with manual branch
naming (not yet supported; for now: clean up locally first).

## Behavior — any other branch

When the current branch is anything other than `master`/`main`:

1. `git add -A` to stage everything.
2. The staged diff is sent to Claude Haiku with the same system prompt; the
   user prompt asks only for `{ "commitMessage": "..." }`.
3. `git commit -F <tmpfile>` records the commit on the current branch.
4. `git push` pushes (no `-u`, since the upstream is presumably set).

## Branch naming convention

`<type>/<kebab-case-slug>` — both lowercase only.

- `<type>` is one of: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`,
  `test`, `build`, `ci`, `chore`, `revert`.
- Slug is 2–6 words, lowercase letters / digits / dashes only.
- Examples: `feat/add-csv-export`, `fix/null-deref-in-parser`,
  `refactor/extract-runner`.

The regex enforced at runtime is `^[a-z]+\/[a-z0-9-]+$`.

## Commit message format

Conventional Commits, drafted by Claude Haiku:

- Subject: `<type>(<optional-scope>): <imperative summary>`, ≤ 72 chars,
  lowercase first letter after the colon, no trailing period.
- Optional body: blank line then 1–4 short bullet points starting with `- `.

## Error modes

| Condition | Behavior |
|-----------|----------|
| `--help` / `-h` | Prints the usage banner, exits 0. |
| Not inside a git repo | Throws `Not a git repository: <cwd>`, exits 1. |
| No pending changes | Logs friendly info message, exits 0. |
| `git add -A` produces an empty staged diff | Logs friendly info, exits 0. |
| `ANTHROPIC_API_KEY` missing/empty | LIVR validation error, exits 1. |
| Anthropic API failure | Underlying error message is surfaced, exits 1. |
| Model returns non-JSON / missing fields | `Model did not return a JSON object` (or field-specific message), exits 1. |
| Suggested branch name fails regex | Throws with the offending name, exits 1. |
| Suggested branch + 9 numeric fallbacks all taken | `SuggestionError` listing every tried name, exits 1. |
| `git checkout -b` / `git commit` / `git push` fails | First 5 lines of stderr are surfaced, exits 1. |

All error paths print via the shared logger (red `✖` prefix) and exit non-zero.

## Diff handling

To keep prompts bounded, the staged diff is truncated to 60 000 characters
before being sent to Claude. A `[diff truncated at 60000 characters]` marker
is appended when truncation kicks in.

## Implementation notes

The script imports its helpers from `src/common/`:

- `logger.ts` — chalk theme, symbols, simple `log` object.
- `command-runner.ts` — `runCommand(command, { cwd, description })` with
  spinner + streamed output.
- `env.ts` — `loadEnv` + `validateEnv<T>(rules)` LIVR helper.
- `errors.ts` — `SuggestionError` class.
- `usage.ts` — styled usage banner renderer.
- `git-utils.ts` — `assertGitRepo`, `hasChanges`, `getCurrentBranch`,
  `isMasterBranch`.

The Anthropic SDK call uses plain JSON-via-prompt (no tool use). The response
is parsed with a small fence-stripping helper that also handles models that
wrap output in ```json fences.
